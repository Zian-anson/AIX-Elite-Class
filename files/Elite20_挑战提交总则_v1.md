# Elite20 挑战提交总则

**适用范围：** Elite20 实验班全部挑战（C2A · C2G · C4A · C4B · C4C · C4D · C5 · C5A · C6 · C6A · C7 · C8 · C9 · C10 · C10H 及后续新增）
**文档性质：** 规范性（normative）。本总则与各挑战 `Cx.md` 并行时，以总则为准；各挑战可在专属章节中加严，不得放松。
**文档结构来源：** 整合 `Elite20_挑战体系说明_最终执行版.md`、`elite20_submission_guide.md`、`拿来主义原则.md`、`Token钱包使用指南_一页纸.md`，并吸收 C2G 挑战在实战中沉淀出的两步审核门模式。
**版本：** v1.3

---

## 0. 一句话规则

> **一次合格的 Elite20 提交 = 可复现的产出 + 可追溯的 AI 日志 + 可验证的拿来说明。**
> 三件缺一，挑战不进入评审。

---

## 1. 核心概念（Domain Model）

本总则以五个领域实体为骨架，挑战流程就是这五个实体沿时间演化的过程。

**1.1 提交包（Submission Package）**
一次提交对应的全部文件集合。由三层组成：
- **通用五件套**（所有挑战必交，见 §3）
- **挑战专章交付物**（各 `Cx.md` 规定的专属文件）
- **演示材料**（视频 / 截图 / 可运行链接，格式不限但必须"评审方在没有你在场时也能理解")

**1.2 审核门（Submission Gate）**
学生与挑战组之间的正式交接点。每经过一道门，学生状态与资源额度都会发生变化。本总则定义三道门（§2），三门职责互不重叠，构成"人审思路 → AI 审证据 → 人审完成"的三明治结构：

- **G1（Proposal Endorsement，人审）** —— 至少一位同伴 + 一位老师背书，确认"这事值得做 / 想清楚了"
- **G2（Evidence Validation，AI 审）** —— G2 Agent / Skill 对证据账本与产物做机械检查，学生需提交"G2 Agent 已通过验证"的报告
- **G3（Completion Endorsement，人审）** —— 至少一位同伴 + 一位老师背书，确认"承诺的事情确实做到了"

**1.3 证据账本（Evidence Ledger）**
一组不可缺少的"过程证据"文件，包括：AI 日志、拿来说明、实验日志 / 训练日志（如适用）、Portfolio 条目。证据账本是**评审能否继续**的二元前置条件，不是给分项。

**1.4 验收标准（Acceptance Criteria）**
分级（Level 1 – Level 4）+ 分维度（功能完整性 · 可复现性 · 学习深度 · 产出质量 · 创新度）。挑战专章可在此基础上对某一维度加权或加严。

**1.5 Portfolio 条目（Portfolio Entry）**
每一次被接受的提交，都要在个人 Portfolio 里落一条记录：挑战编号、达到的 Level、使用的 Token / 算力、核心产出链接、关键反思。Portfolio 是 Elite20 对外输出的主要简历单位。

## 2. 通用流程（L0 → L7 状态机）

```
L0 Intent           学生在群里一句话承诺 → 记录
  │
L1 Proposal         学生撰写「方案草案」→ 找 ≥1 同伴 + ≥1 老师沟通
  │ G1: Proposal Endorsement (同伴 + 老师书面背书)
L2 Resource         资源预估 ≤ ¥70  → 直接开跑（无需额外审批）
                    资源预估 > ¥70  → 需 Dr. Xiao 签字批复 → 发放 Token / 算力券
  │
L3 Execution        学生开跑（GitHub repo / Skill dev / 实验）
                    执行期若累计消耗 > ¥70 → 暂停 → 回 L2 补批复 → 继续
  │
L4 Submission       学生提交完整 Submission Package → 跑 G2 Agent
  │ G2: Evidence Validation (AI Agent/Skill 机械审查)
L5 Completion Review 提交产出 + G2 验证报告 → ≥1 同伴 + ≥1 老师复核
  │ G3: Completion Endorsement (同伴 + 老师书面签收)
L6 Iterate / Accept 通过 → L7；未通过 → 回 L4 / L3 / L1（视原因）
  │
L7 Portfolio        产出入 Portfolio；Token 消费结账；公示
```

**L2 资源审批说明：**

Token 钱包与算力券**不是强制领取**。绝大多数挑战都能用自有账号或免费额度跑完 —— 直接进入 L3 开干，不走任何资源审批流程。**这是默认路径，也是绝大多数同学的路径**。

**只在一种例外情况下需要额外审批：** 当挑战需要**超出常规 API 基线**的资源时，合计预估 > **¥70 RMB** 才触发 Dr. Xiao 签字批复流程。典型场景包括：

- **高成本多模态 / 视频生成** —— 例如 CDS 2.0、豆包 Seedance 这类单 token 成本高、生成一分钟视频动辄数十美元的服务
- **长周期大模型训练 / 微调** —— 需要多小时甚至多天 GPU 时间的 fine-tuning、RLHF、或 C2G 这类训练竞赛
- **付费 API 加购包** —— 在默认 API 额度之外另行申请的高级 SKU、专属模型访问、或批量调用配额

如果你做的是**普通 GitHub 项目**（用 Claude / GPT-5 API 写代码、做 demo、跑小规模实验），绝大多数情况都在 ¥70 以下，不触发这道门——放心直接做。Dr. Xiao 批复这道门是为**进阶项目**准备的"资源放大器"，不是为常规项目准备的"摩擦检查"。

**阈值选定理由：** ¥70 大致对应一次中等强度 Claude Opus / GPT-5 长对话，或一次 A100 小时级小实验的成本。低于这个数，任何同学用自己的 API 额度或免费赞助都能搞定；高于这个数，开始影响预算盘子，需要有人对"值不值得花"做二次判断。

**Dr. Xiao 批复的承载物：** `姓名_CX_Xiao批复.md`（见 §3 条件件）。不触发阈值时此文件可省略 —— 不存在不扣分。

**执行期越过阈值的处理：** 如果你起初预估 ≤ ¥70、但跑到一半发现要超，**必须暂停**，带着实际消耗数据回 Dr. Xiao 申请追加批复；**不可**先花再报。未经批复的超额消耗由学生自行承担，不进入 Portfolio 的 Token 结账记录。

**为什么这么设计（三明治结构 "人–AI–人"）：**

- **G1 是社会化承诺。** 把方案想清楚的成本很低，但找到愿意给你背书的同伴和老师的成本较高——这个过程本身就逼学生把草案讲到别人能听懂的程度。Token / 算力不是凭一纸草案换来的，是凭两个人愿意为你的想法署名换来的。
- **G2 是机械化合规。** 证据账本齐不齐、文件名规范不规范、复现脚本能不能跑、License 有没有标、每个章节模板填没填——这些都是规则化、可自动判定的检查项，正适合交给 AI Agent / Skill，不占用人的注意力。
- **G3 是社会化验收。** 产出能否被接受，最终仍由人判断。但评审人拿到的已经是经过 G2 扫描的干净包，可以把全部注意力放在"产出质量 / 创新度 / 学习深度"的人类判断项上。

**G1 / G2 / G3 三道门的职责互不重叠：**

| 门 | 审方 | 审什么 | 不审什么 | 输出物 |
|---|---|---|---|---|
| G1 | ≥1 同伴 + ≥1 老师 | 方向是否值得做、想清楚了没有 | 代码、实现细节 | 两份背书记录（见 §6.2） |
| G2 | G2 Agent / Skill（AI） | 证据账本齐全度、规范合规、可复现性机械检查 | 产出的创意与深度 | `姓名_CX_G2验证报告.md` |
| G3 | ≥1 同伴 + ≥1 老师 | 承诺的事情是否真的做到了、达到哪一级 | 证据齐不齐（G2 已把关） | 两份完成签收记录 + Level 判定 |

**一个约束：** G1 的同伴与 G3 的同伴**不必**是同一人（鼓励不同角度审视）；但 G1 的老师与 G3 的老师**建议是同一人**，以形成"开始时为你背书 → 完成时为你签收"的连贯责任链。

## 3. 通用交付物清单（六件套）

**所有挑战的提交包都必须包含以下六件。命名统一：`姓名_CX_xxx.md`。**

| # | 文件 | 作用 | G 门位置 |
|---|---|---|---|
| 1 | `姓名_CX_方案草案.md` | 在拿到资源前说清"打算做什么 / 凭什么觉得能做成 / 失败了怎么办" | G1 输入 |
| 2 | `姓名_CX_G1背书.md` | 收录同伴与老师对方案草案的书面背书（见 §6.2 模板） | **G1 产物** |
| 3 | `姓名_CX_方案设计.md` | 在方案草案基础上扩展：目标 Level、技术选型、实验矩阵或模块分解 | 交付时 |
| 4 | `姓名_CX_AI日志.md` | 全程与 AI 的关键对话、让 AI 做了什么、自己做了什么、为什么 | **G2 必扫项** |
| 5 | `姓名_CX_拿来说明.md` | 复用了哪些开源 / 历史作品 / 他人方案，改了什么、为什么 | **G2 必扫项** |
| 6 | `姓名_CX_G2验证报告.md` | G2 Agent / Skill 生成的机械验证报告；必须包含 `APPROVE` 标记才能进入 G3 | **G2 产物** |
| 7 | `姓名_CX_Portfolio条目.md` | 一页纸：挑战编号、Level、Token 消耗、产出链接、关键反思 | L7 |

**条件件（仅在资源预估 > ¥70 时必交）：**

| # | 文件 | 作用 | G 门位置 |
|---|---|---|---|
| C1 | `姓名_CX_Xiao批复.md` | Dr. Xiao 对超阈值资源预算的签字批复记录；格式见 §6.4 | L2 条件项 |

**硬约束：**
- G2 Agent 验证报告里任一 `FAIL` 项未解决 → 提交不得进入 G3。
- G1 背书不足（少于 1 同伴 + 1 老师）→ 资源不发放 / 不得继续，不得进入 L2。
- 资源预估 > ¥70 但缺 `Xiao批复.md` → Token / 算力券不发放；学生仍可选择用自有资源在 ≤ ¥70 预算内推进。

**方案草案必答的四个问题**（C2G 模式的推广，所有挑战适用）：
1. 你打算做什么？解决什么具体问题？
2. 你凭什么觉得这个方向有效？（引用 ≥ 1 篇论文 / ≥ 1 个开源项目 / ≥ 1 条前人经验）
3. 你拿到的 Token / 算力打算怎么花？预估跑几次实验、每次看什么？
4. 失败了怎么办？结果不如预期时你的下一步是什么？

## 4. 通用验收标准（Acceptance Criteria）

**4.1 Level 语义（所有挑战共用）**

| 级别 | 含义 | 典型信号 |
|---|---|---|
| 🥉 Level 1 | 跑通 / 复现基线 | 能复现官方/前人最简版本 |
| 🥈 Level 2 | 单点改进 | 做一个变量的对照实验，有正向收益 |
| 🥇 Level 3 | 组合优化 / 工程化 | 多个改进组合、产出可被他人直接使用 |
| 💎 Level 4 | 挑战 SOTA / 原创贡献 | 全球可见的排名 / 论文级别的产出 / 进入职业通道的信号 |

各挑战的专章可把 Level 映射到具体指标（如 C2G：BPB；C5：项目 star / 真实用户数；C4B：公众号阅读量）。

**4.2 评分维度（每维 0–4 分，G3 使用）**

- **功能完整性** —— 按方案设计，承诺的是否都做了
- **可复现性** —— 评审方按 README / 复现脚本是否能独立跑通
- **学习深度** —— AI 日志与拿来说明是否显示真实的思考，而不是机械拼接
- **产出质量** —— 代码 / 文档 / 内容本身的工程度与美感
- **创新度** —— 相对于现有方案的增量贡献

**4.3 挑战总分与 Level 挂钩**
- 总分 ≥ 10 → Level 1
- 总分 ≥ 14 且可复现性 ≥ 3 → Level 2
- 总分 ≥ 17 且创新度 ≥ 3 → Level 3
- 总分 ≥ 19 且全球可见信号满足 → Level 4

具体分数阈值各挑战可微调，原则"严进严出"。

## 5. GitHub 项目型挑战专章（以 C5 为示范）

本章是所有"学生交付物主要载体是一个 GitHub 仓库"的挑战的横切规范。适用 C5、C2G、C4B、C4C、C5A、C6A 及任何后续以 Repo 为交付形态的挑战。

### 5.1 Repo 结构规范（必需目录与文件）

```
{username}-{challenge-id}-{project-name}/
├── README.md                    ← 必填，见 §5.2
├── AI_LOG.md                    ← 证据账本之一；可同名 姓名_CX_AI日志.md
├── REUSE.md                     ← 证据账本之二；可同名 姓名_CX_拿来说明.md
├── PROPOSAL.md                  ← 方案设计（方案草案可单列或合并）
├── CHANGELOG.md                 ← 每次有意义的提交留一条
├── LICENSE                      ← 强烈推荐 MIT / Apache-2.0
├── requirements.txt / pyproject.toml / package.json
├── src/                         ← 源码
├── tests/                       ← 至少一个 smoke test
├── examples/                    ← 可运行的 demo
├── scripts/
│   └── reproduce.sh             ← 一条命令复现核心结果
└── docs/
    └── demo.md / demo.mp4 链接   ← 演示材料
```

**挑战专章可以加文件但不得删除必需项。**

### 5.2 README.md 必备段落

README 是评审方看到你作品的"第一面"，必须用**一分钟之内**让人明白：

1. **一句话定位**（这是什么，为谁做的）
2. **演示入口**（在线链接 / GIF / 30 秒视频）
3. **快速上手**（≤ 5 行命令，复制就能跑）
4. **效果数据**（Level 1 / 2 / 3 / 4 对应的硬指标）
5. **证据账本入口**（AI_LOG / REUSE / PROPOSAL 的链接）
6. **致谢与拿来**（清楚列出复用的项目、论文、前辈作品）

**反例（会直接被 G2 退回）：**
- README 只写了 "My C5 project"
- 没有演示入口
- 没有复现命令
- 没有链接到 AI_LOG / REUSE

### 5.3 提交规范（Commit / PR / Issue）

- **Commit message** 使用祈使语气，前缀标签：`feat: … / fix: … / docs: … / refactor: … / chore: …`
- **PR**（如果协作）必须填模板：问题 / 方案 / 测试证据 / 拿来说明（如引入了新依赖）
- **Issue** 用于记录遇到的坑与决策点；挑战评审会翻 Issue。

### 5.4 复现性硬要求

`scripts/reproduce.sh` 应做到：
- 一条命令跑完核心结果（`bash scripts/reproduce.sh`）
- 在评审方完全陌生的机器上（符合 README 环境要求）可成功跑通
- 如果有随机性，固定 seed 并在日志中打印

**不可复现 → 可复现性维度直接 0 分，Level 2 封顶。**

### 5.5 演示材料

至少满足以下一项：
- 部署到公网的可访问链接（Vercel / HuggingFace Spaces / 自部署都可）
- 一段 ≤ 3 分钟的演示视频（链接到 README 顶部）
- 可运行的 colab notebook

**产品/内容类挑战（C4B 公众号、C4A 社群运营）** 的"演示材料"是真实发布链接 + 至少一项可观测指标（阅读数 / 转化数 / 用户反馈）。

## 6. 审核机制（Review Workflow）

**6.1 审核角色（按门分配）**

| 门 | 主审人 | 人数下限 | 备注 |
|---|---|---|---|
| G1 | 同伴（同届学员） + 老师（挑战组 / 方向导师） | 各 ≥ 1，合计 ≥ 2 | 同伴负责"听得懂吗 / 合理吗"；老师负责"值不值得做 / 难度合不合适" |
| L2 资源超阈值审批（条件触发） | **Dr. Xiao** | 1 | **仅**在资源预估 > ¥70 时触发；≤ ¥70 无需任何审批 |
| G2 | G2 Agent / Skill（AI） | 1 | 由 Elite20 官方维护（`elite20-submission-validator` skill，默认配置见 §6.4） |
| G3 | 同伴（同届学员） + 老师（挑战组 / 方向导师） | 各 ≥ 1，合计 ≥ 2 | Level 3 / 4 的挑战**必须**由方向导师参与 G3；Level 1 / 2 由挑战组老师即可 |

**同伴背书不接受互签**：A 给 B 背书，不代表 B 自动可给 A 背书 —— 每一次背书都要有实质对话与独立判断，可被挑战组抽查。

**G1 / G3 所有人审发起必须在 Elite20 微信群公开发起**：申请人在微信群里 @ 具体的同伴与老师，粘贴方案草案 / 最终提交链接、标注"请求 G1 背书" 或"请求 G3 签收"。三点目的：

1. **留痕可追溯** —— 背书请求与响应都有群聊记录，事后审计可回溯。
2. **防止私下串签** —— 公开发起自然抑制"只找好说话的人"的逆向选择；群里所有人都看得到谁在给谁背书。
3. **创造学习外部性** —— 即使不是背书人，其他同学读到你的方案草案也在学"一份可被背书的方案长什么样"，读到别人的完成提交也在学"什么叫达到 Level X"。

**群里消息最小格式（申请人发出）：**

```
#挑战背书请求 CX / G1
@同伴某某 @某老师
请求 G1 背书（方案草案）
草案链接: ...
目标 Level: X
关键判断点（希望你们重点看）:
  - ...
  - ...
```

```
#挑战签收请求 CX / G3
@同伴某某 @某老师
请求 G3 签收（完成提交）
仓库链接: ...
G2 验证报告: APPROVE (附在仓库 G2_VALIDATION.md)
声称达到的 Level: X
```

**背书人的回复也必须在群里可见**（可选：在群里回"已看完，填完背书表放在仓库"指向具体文件链接；或直接把背书段落粘在群里再落盘到仓库）。**仅私聊 / 邮件 / 线下达成的背书不被接受。**

**Dr. Xiao 的 L2 资源批复不走群**（涉及具体金额，用微信/邮件私下沟通更合适），但批复落盘为 `Xiao批复.md` 后，学生应在群里发一条"已获 Dr. Xiao 对 CX 的资源批复，上限 ¥X"作为公示，便于大家知道预算与风险。

**资源审批不在主流程上**：L2 资源审批是**可选侧支**，不是第四道门。默认路径是 G1 → L3，直接开跑。只有当预算跨过 ¥70 阈值，才绕道经过 Dr. Xiao 的签字环节。这是为了把流程摩擦留给真正需要决策的场景。

每一位背书人（同伴或老师）各自填一份并交给学生汇总到 `姓名_CX_G1背书.md` 或 `姓名_CX_G3签收.md`：

**6.4 Dr. Xiao 资源批复模板（仅在资源预估 > ¥70 时使用）**

学生把方案草案 + 资源预算表交给 Dr. Xiao（可通过微信 / 邮件 / 线下），Dr. Xiao 若同意则回复一段可复制的批复文本，学生落盘为 `姓名_CX_Xiao批复.md`：

```
挑战: CX
申请人: [姓名]
提交时间: {ISO-8601}
预估资源：
  - Token 消耗: ¥{A}  (含模型明细: GPT-5 x次, Claude x次, ...)
  - 算力: ¥{B}  (含硬件明细: H100/A100/云服务商 x小时)
  - 合计: ¥{A+B}
附加说明（学生填写）:
  - 为什么需要超过 ¥70 的预算：...
  - 不超 ¥70 行不行：...
  - 失败了这笔钱的学习价值是什么：...

—— 以下为 Dr. Xiao 批复 ——
批复: APPROVE / APPROVE_WITH_CAP / REJECT
上限: ¥{X}（若 APPROVE_WITH_CAP）
附加条件（若有）: ...
备注: ...
签字 & 日期: Dr. Xiao, {YYYY-MM-DD}
```

**三种可能的批复：**
- **APPROVE** —— 按申请金额批复，学生可领取全额资源
- **APPROVE_WITH_CAP** —— 批一个比申请金额更低的上限，学生须在该上限内收敛方案或自补差额
- **REJECT** —— 不同意超阈值投入；学生可选择"改方案压到 ¥70 以内"或"自费"或"放弃挑战"

**6.5 决定的语义**
- **APPROVE / ENDORSE** —— 进入下一门或接受提交
- **REQUEST_CHANGES** —— 方向对，补齐指定项后重审（同一审核人，不重新排队）
- **REJECT** —— 方向有根本性问题，回 L1 重写方案草案（仅 G1 / G3 人审有 REJECT 权；G2 AI 只会 APPROVE 或 REQUEST_CHANGES；Dr. Xiao 的 REJECT 只针对资源投入，不否定挑战本身）

## 7. 拒绝与复审流程（Reject & Resubmission）

**7.1 被 G1 未背书的处方**
找不到同伴或老师愿意背书，说明方案草案讲得不够清楚或方向有问题。**不进入资源发放**。学生应回到 L1 改稿、另寻讨论对象。无上限次数，不产生"失败记录"。

**7.2 被 G2 退回的唯一处方**
按 G2 Agent 报告里的 `FAIL` 清单逐项修复 → 重跑 G2 Agent → 拿到 `APPROVE` 版本。无上限次数。被退回不扣分，不产生"失败记录"。

**7.3 被 G3 判定未达预期 Level 的处理**
- 如未达承诺的 Level 但达到了次一级 Level → 按实际 Level 接受，Portfolio 条目如实记录。
- 如未达任何 Level → REQUEST_CHANGES，指向具体维度补强。
- 连续两轮仍未达任何 Level → REJECT，回 L1 重写方案草案。这种情况不视为失败，但 Token 消耗与时间投入如实记入 Portfolio。

**7.4 申诉**
- **对 G2 AI 判定的申诉** —— 学生若认为 G2 Agent 的某个 `FAIL` 属于误判（例如模板更新滞后、Agent 未识别某个等价实现），可请求**一位老师**以书面形式 override，override 记录附在 `G2验证报告.md` 末尾；老师 override ≥ 2 个检查项时需挑战组长复核。
- **对 G3 人审打分的申诉** —— 学生可要求方向导师复核。复核结果为终判。

---

## 8. 三条红线（Red Lines）

**违反任何一条 → 提交不被接受，Token 已消耗不返还。**

1. **AI 日志缺失 / 伪造**
   没有 AI 日志，或日志与产出明显不符（例如声称"AI 帮我写了架构"但 Git 历史里没有对应的对话节点 / 设计迭代）。
2. **伪拿来**
   REUSE.md 未列出实际复用的项目 / 代码 / 内容；或 License 违规（把 AGPL 作品装进 MIT 仓库，把他人作品说成自己原创）。
3. **不可复现**
   评审方按 README 与 `reproduce.sh` 无法独立跑通核心结果，且作者不响应 / 不修复。

## 9. 自检工具：elite20-g2-validator Skill

**`elite20-g2-validator.skill`** 是 Elite20 官方提供的 G2 自检工具。学生在把提交交给 G3 同伴与老师之前，**应该**先用这个工具把提交包跑一遍，直到拿到 `APPROVE` 再约背书。

### 为什么提供这个

G2 是机械合规关卡 —— 它审的东西（文件是否存在、README 是否齐全、AI 日志是否足够多决策节点、复现脚本有没有 shebang、etc.）全部是**规则化、可自动判定**的。把这类工作交给 AI Agent 而不是人审，有三个理由：

1. **省老师时间** —— 老师的注意力应该花在 G1（值不值得做）和 G3（做得好不好）这两个需要判断力的门上，不该浪费在"你的 README 少一个段落"这种问题上。
2. **省学生时间** —— 你不用每次都问"我现在到底能不能交"；打开 skill 跑一下就知道，分钟级反馈。
3. **统一标准** —— 不同老师看同一份提交，可能对"AI 日志算不算够"有不同感受。AI 验证器按同一套规则判定，所有人的 G2 门槛一致。

### 学生怎么用

**第一步：安装 skill。** 在 Elite20 群文件里下载 `elite20-g2-validator.skill` 文件，在你的 Claude Code / Claude 应用里安装。

**第二步：在你本地跑。** 准备好你的提交包（通常是一个 GitHub repo 的本地 clone 或一个文件夹），对 Claude 说：

> "跑一下 G2 验证 /path/to/my/c5-submission"

或者

> "Can you run G2 on my C5 submission at `~/projects/my-c5-repo`?"

Claude 会自动调用 skill，跑完十项必检 + 两项条件检，把结果写成 `G2_VALIDATION.md` 放在你的提交包里，并用大白话告诉你哪里 FAIL、怎么修。

**第三步：迭代到 APPROVE。** 按 `FAIL` 行里的 `fix_hint` 改 → 重新跑 → 再改 → 再跑。直到报告顶部写 `Overall Decision: APPROVE`。

**第四步：带着 APPROVE 报告进 G3。** 把 `G2_VALIDATION.md` 连同整个仓库一起发给 G3 的同伴 + 老师。评审方看到顶部的 `APPROVE` 就知道证据关已过，可以专注看产出质量。

### G2 Agent 检查的十项必检（复述自 §6.3）

| ID | 检查项 |
|---|---|
| G2-01 | 方案草案存在且 ≥500 字、覆盖四个关卡问题 |
| G2-02 | G1 背书 ≥1 同伴 + ≥1 老师，均 ENDORSE |
| G2-03 | AI 日志 ≥3 个关键决策节点 |
| G2-04 | 拿来说明每项都标注 License |
| G2-05 | README 六段必备内容齐全（GitHub 类） |
| G2-06 | scripts/reproduce.sh 存在且可解析 |
| G2-07 | Smoke test ≥ 1 且能被识别 |
| G2-08 | 命名规范统一（CJK 或 English，不混用） |
| G2-09 | 三条红线自测覆盖齐全 |
| G2-10 | Level 对应的硬指标在 README 可见 |

**两项条件检**（仅在满足触发条件时激活）：

- **G2-C1** Dr. Xiao 批复 —— 仅当资源预估 > ¥70 触发（见 §2 L2 说明）
- **G2-C2** 微信群消息留痕 —— 每位背书人都要有群消息 URL / 截图 / 时间戳

### 不能做什么

- **不能手写 `G2_VALIDATION.md`。** 这份报告的权威性来自它是机械产出的——学生手写的 `APPROVE` 报告会在 G3 被立即识破（文件头的 validator 版本号、时间戳、check 结构都对不上）。
- **不能用 skill 的 `APPROVE` 结果替代 G1 / G3 的人审背书。** G2 通过只意味着"证据齐全 / 格式合规"，不意味着"想法有价值 / 做得好"。后两者必须人来判。
- **不能在本地改 skill 的检查规则自己放水。** skill 有版本号并嵌在报告里，挑战组会在抽查时比对。动过的 skill 等同于伪造。

### 何时不用这个工具

如果你的挑战**不是** GitHub 项目型（例如 C4B 公众号发布、C4A 社群运营），当前 v0.1 版本的 validator 会把 README / reproduce / smoke-test 三项标为 `N/A`——剩余的普适检查仍然有效。后续 v0.2 会针对其他挑战类型提供专用检查包。

### 申诉路径

如果你相信某一个 FAIL 是误判（例如 skill 的文件名模式没识别到你的等价命名），按 §7.4 找一位老师以书面 override 覆盖单独一项即可——整个报告仍然以 APPROVE 进入 G3，override 记录附在报告尾部。请**不要**通过手改报告绕过。
